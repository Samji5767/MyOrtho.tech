# ADR-002: PostgreSQL via Direct pg Pool (No ORM)

## Status

Accepted

## Date

2026-01-15

## Context

MyOrtho.tech stores and queries clinical data: patient records, treatment plans, case workflows, manufacturing orders, and billing. This domain is:

- **Query-heavy and join-intensive.** Clinical queries often span multiple tables (patient → case → treatment plan → appliance → manufacturing order).
- **Performance-sensitive.** Report generation and dashboard aggregations benefit from hand-tuned SQL with appropriate indexes and CTEs.
- **Auditable.** SQL queries executed against patient data must be deterministic and reviewable. Magic generated by an ORM is harder to audit in a compliance context.
- **Already schema-managed.** The database schema was established through numbered migration files; we do not need an ORM to manage schema.

## Decision

We use the `pg` npm package with a connection `Pool` injected via NestJS dependency injection using the custom token `PG_POOL`.

All queries use parameterized placeholders (`$1`, `$2`, ...) — never string interpolation or template literals. The pool is configured in `DatabaseModule`, which is marked `@Global()` so any feature module can inject `PG_POOL` without re-importing `DatabaseModule`.

Example injection pattern:

```typescript
@Injectable()
export class CaseService {
  constructor(@Inject('PG_POOL') private readonly pool: Pool) {}

  async getCaseById(id: string): Promise<Case> {
    const { rows } = await this.pool.query(
      'SELECT * FROM cases WHERE id = $1 AND deleted_at IS NULL',
      [id],
    );
    return rows[0] ?? null;
  }
}
```

## Consequences

### Positive

- Full SQL control: CTEs, window functions, lateral joins, and `EXPLAIN ANALYZE` are trivially available
- No ORM abstraction leaks: the SQL executed is exactly what was written, making audits and performance reviews straightforward
- Easy to optimize: query tuning is a matter of editing the SQL string, not fighting an ORM's query generator
- Minimal dependency surface: `pg` is a stable, low-level driver with predictable behavior

### Negative

- No automatic migration generation: schema changes require manually authored SQL migration files
- No type-safe query builder: typos in column names are caught at runtime (or in tests), not at compile time. Mitigated by thorough service-level tests and a typed result mapping layer
- Manual schema management: developers must maintain migration files and ensure they are idempotent

## Alternatives Considered

### TypeORM

**Rejected.** TypeORM's decorator-based entity coupling (`@Entity`, `@Column`, `@ManyToOne`) ties the domain model tightly to the persistence layer. Generated queries are often suboptimal for complex clinical aggregations, and the eager/lazy loading model introduces subtle N+1 risks. Decorator metadata also complicates testing.

### Prisma

**Rejected.** Prisma's `schema.prisma` as the single source of truth for schema creates drift risk when the database already has a migration-managed schema. Mapping between the Prisma schema and existing migration files during onboarding introduces synchronization overhead and potential for divergence. Prisma's generated client is also a heavyweight artifact to vendor.

### Knex (Query Builder)

**Considered but rejected.** Knex provides a programmatic query builder that sits between raw SQL and a full ORM. While it reduces typo risk for simple queries, the query builder DSL adds cognitive overhead for complex SQL (CTEs, recursive queries, custom aggregates) that must be expressed through escape hatches anyway. The benefit does not outweigh the added dependency and learning surface for the team.
