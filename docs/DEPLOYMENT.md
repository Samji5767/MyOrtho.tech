# Deployment Guide — MyOrtho.tech v2.0.0-rc1

VPS deployment with Docker Compose and nginx. This guide covers a fresh install on a single host.

---

## Prerequisites

| Requirement | Minimum |
|---|---|
| Docker | 24.0+ |
| Docker Compose | v2 (plugin, not standalone) |
| RAM | 4 GB (8 GB recommended with AI engine) |
| Disk | 20 GB free (scans and exports grow over time) |
| OS | Ubuntu 22.04 LTS or Debian 12 |
| Domain | A/AAAA record pointing to host IP, TLS certificate issued |

Verify before proceeding:

```bash
docker --version          # Docker version 24.x.x
docker compose version    # Docker Compose version v2.x.x
```

---

## Step 1 — Clone the Repository

```bash
git clone https://github.com/your-org/myortho-tech.git /opt/myortho
cd /opt/myortho
git checkout v2.0.0-rc1
```

---

## Step 2 — Configure Environment

Copy the template and open it for editing:

```bash
cp .env.example .env
nano .env
```

### Required secrets

Generate strong random values for JWT_SECRET and INTERNAL_API_SECRET:

```bash
openssl rand -hex 32   # run twice — once for JWT_SECRET, once for INTERNAL_API_SECRET
```

Set ENCRYPTION_KEY to any 32-character string used exclusively for PHI encryption. **Back this value up before starting the stack.** Loss of this key means permanent, irrecoverable loss of all encrypted patient data.

```bash
# Example — replace with a value generated by your own key management process
ENCRYPTION_KEY=a7f3d92e1b4c8f6a0d5e2b9c7f1a4e8d
```

### Minimum required values to edit in .env

```dotenv
# Database
POSTGRES_PASSWORD=<strong random password>
DATABASE_URL=postgresql://myortho_admin:<strong random password>@database:5432/myortho_tech

# Security
JWT_SECRET=<openssl rand -hex 32>
ENCRYPTION_KEY=<32-character key — back up separately>
INTERNAL_API_SECRET=<openssl rand -hex 32>

# Bootstrap admin (first boot only — can be removed after first start)
MYORTHO_ADMIN_EMAIL=admin@your-domain.com
MYORTHO_ADMIN_PASSWORD=<strong password, min 12 chars>
MYORTHO_ADMIN_NAME=Platform Admin

# Frontend origin
FRONTEND_URL=https://your-domain.com
CORS_ORIGIN=https://your-domain.com
NEXT_PUBLIC_API_URL=https://your-domain.com
```

### Optional integrations

```dotenv
# Stripe billing (leave blank to run in mock mode)
STRIPE_SECRET_KEY=sk_live_...
STRIPE_WEBHOOK_SECRET=whsec_...

# AI Copilot
COPILOT_LLM_PROVIDER=openai
COPILOT_LLM_API_KEY=sk-...
COPILOT_EMBED_API_KEY=sk-...

# Treatment plan AI
TREATMENT_PLAN_AI_URL=http://your-ai-endpoint:8001

# AI segmentation fallback (when no model checkpoint)
SEGMENTATION_FALLBACK_ENABLED=true
```

## Step 3 — Build and Start

```bash
docker compose up -d --build
```

Services started:
- `database` — PostgreSQL 15, internal network only (not exposed to host)
- `redis` — Redis 7, internal network only
- `migrate` — one-shot migration runner; exits 0 when all migrations complete
- `backend` — NestJS API on port 4000
- `frontend` — Next.js static export served on port 3005
- `ai-engine` — FastAPI AI service on port 8000 (optional)

Watch migration progress:

```bash
docker compose logs -f migrate
```

The backend waits for `migrate` to exit 0 before it starts. Do not skip this step.

---

## Database Migrations

Migrations run automatically via `database/migrate.sh` on every `docker compose up`. They are idempotent and safe to re-run. Key migration files (in `database/migrations/`):

| File | Purpose |
|---|---|
| `000_db_init.sql` | Schema foundation, extensions |
| `001_auth_users.sql` | Users, roles, sessions |
| `002_vps_core_schema.sql` | Core clinical entities |
| `003_phases_15d_15e_15f.sql` | Case lifecycle state machine |
| `009_phase_21.sql` | Performance indexes |
| `021_performance_indexes.sql` | Additional query indexes |
| `024_phase_39_49.sql` | Feature flags, audit log |
| `027_phase_86_100.sql` | FHIR export, manufacturing queue |
| `028_phase_23_treatment_planning.sql` | Treatment plan stages |
| `029_phase_25_production_validation.sql` | Production-safety constraints |

To view applied migrations:

```bash
docker compose exec database psql -U myortho_admin -d myortho_tech \
  -c "SELECT filename, applied_at FROM schema_migrations ORDER BY applied_at;"
```

## nginx Configuration

nginx acts as a reverse proxy in front of all three application ports. Place the following in `/etc/nginx/sites-available/myortho`:

```nginx
server {
    listen 80;
    server_name your-domain.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name your-domain.com;

    ssl_certificate     /etc/letsencrypt/live/your-domain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/your-domain.com/privkey.pem;

    # Frontend (Next.js static export)
    location / {
        proxy_pass http://127.0.0.1:3005;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Backend API
    location /api/ {
        proxy_pass http://127.0.0.1:4000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        client_max_body_size 200M;   # STL uploads can be large
    }

    # AI engine (restrict to internal if not needed externally)
    location /ai/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
    }
}
```

Enable and reload:

```bash
ln -s /etc/nginx/sites-available/myortho /etc/nginx/sites-enabled/
nginx -t && systemctl reload nginx
```

## Health Checks

After startup, verify each layer:

```bash
# Backend API health
curl -s https://your-domain.com/api/health | jq .

# Session endpoint (should return 401 when not logged in — that's correct)
curl -s -o /dev/null -w "%{http_code}" https://your-domain.com/api/auth/session

# Version
curl -s https://your-domain.com/api/version | jq .

# AI engine (if running)
curl -s http://127.0.0.1:8000/health | jq .
```

Check all container statuses:

```bash
docker compose ps
```

All services except `migrate` (which exits 0) should show `Up (healthy)` or `Up`.

## Rollback Procedure

```bash
# 1. Stop the running stack
docker compose down

# 2. Check out the previous release tag
git checkout v1.0.0-rc2

# 3. Rebuild and start
docker compose up -d --build
```

**Note on database rollback:** Migrations are applied forward only. If the new release introduced schema changes that are incompatible with the previous code version, restore from a pre-upgrade pg_dump backup rather than relying on git checkout alone. See BACKUP_RESTORE.md for restore procedure.

## Production Checklist

- [ ] POSTGRES_PASSWORD changed from default
- [ ] JWT_SECRET set to 64+ hex characters
- [ ] ENCRYPTION_KEY backed up to a secure secrets manager
- [ ] INTERNAL_API_SECRET set
- [ ] MYORTHO_ADMIN_EMAIL / PASSWORD set before first boot
- [ ] FRONTEND_URL and CORS_ORIGIN set to production domain
- [ ] NODE_ENV=production
- [ ] TLS certificate installed and nginx config tested
- [ ] Firewall: only ports 80, 443, and 22 open to the internet
- [ ] Docker socket not exposed
- [ ] Daily pg_dump cron configured (see BACKUP_RESTORE.md)
